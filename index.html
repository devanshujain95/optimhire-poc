Chrome Extension to mobile

Reuse strategy

The goal is not to copy a Chrome Extension directly into mobile. The right approach is to preserve the reusable automation core and rebuild the platform execution layer for Android and iOS.

Reuse directly

  • TypeScript utility functions
  • Business rules and job board metadata
  • Field mapping and question/answers matching
  • Existing JSON structures and backend API clients
  • Validation helpers and normalized automation outputs

Adapt for mobile

  • DOM parsing timing inside React Native WebView
  • Bridge messaging between page JS/TS & app state
  • Resume upload and native permissions
  • Manual fallback states for unsupported fields

Implementation roadmap

How I would take this from POC to production

  1. Audit the Plasmo Chrome Extension Review the already created extension, separate reusable JS/TS logic from browser-only APIs, and document mobile limitations.
  2. Create a shared automation core Move field mapping, question matching, JSON models, API clients, and site rules into a mobile safe shared package.
  3. Build the React Native WebView layer Implement scan, classify, fill, skip, focus, reset, and review commands through a reliable WebView bridge.
  4. Integrate profile, resume, and backend APIs Connect authentication, user profile data, resume handling, existing APIs, and normalized application workflow state.
  5. Scale across job boards with QA Add site adapters incrementally, test field types and edge cases, and ship Android/iOS builds with deployment documentation.
